## 1. 问题与机会

### 1.1 问题陈述

**有特殊饮食需求的人群（减肥、备孕、糖尿病、过敏等）每天面临"吃什么"的决策疲劳，现有的美食推荐和菜谱平台无法将"个人健康需求 × 用餐场景 × 一键采购"三个环节串联起来，导致用户需要跨越 3-5 个 APP 才能完成"从想吃到吃上"的完整闭环。**

### 1.2 用户痛点

| 痛点 | 具体表现 | 当前替代方案 | 频率 |
|------|---------|-------------|------|
| 决策疲劳 | 打开外卖/菜谱APP不知道吃什么，翻 20 分钟无果 | 随便点常吃的，营养不均衡 | 每日 1-2 次 |
| 需求匹配差 | 减肥人群找不到真正低卡的周边美食；备孕人群不知道哪些餐厅有叶酸丰富的菜品 | 手动搜索关键词，自行判断 | 每周 3-5 次 |
| 采购链路断裂 | 看到菜谱想自己做，需要手动记下食材 → 打开买菜APP → 逐个搜索加入购物车 | 截图或手抄清单 | 每周 2-3 次 |
| 信息碎片化 | 健康饮食知识分散在小红书、抖音、公众号，不成体系 | 收藏夹吃灰 | 持续 |

### 1.3 市场机会

- **健康饮食市场**：2025 年中国健康食品市场规模超 8000 亿元，年增速 15%+
- **即时零售渗透**：多多买菜、叮咚买菜、美团买菜日活合计破亿，生鲜即时配送基础设施成熟
- **AI 个性化能力**：大模型能力使得"千人千面的饮食推荐"从概念变为可行
- **政策红利**："健康中国 2030"推动全民健康意识提升，减重、慢病管理需求爆发

### 1.4 竞品格局

| 竞品 | 优势 | 缺失 |
|------|------|------|
| 下厨房/豆果美食 | 菜谱内容丰富 | 无个性化健康需求匹配、无一键购 |
| 美团/饿了么 | 周边餐饮覆盖广 | 无法按健康需求筛选、无居家场景 |
| Keep/薄荷健康 | 饮食记录+热量管理 | 不解决"去哪吃/怎么做" |
| 叮咚买菜/盒马 | 生鲜供应链强 | 无菜谱推荐、无个性化需求匹配 |
| **真享吃定位** | **健康需求 → 智能推荐 → 一键购买的完整闭环** | — |

## 2. 解决方案概述

### 2.1 产品一句话

**真享吃是一个基于个人健康需求的智能美食推荐平台——告诉你要吃什么、去哪吃、怎么做，并且一键把食材送到家。**

### 2.2 核心价值主张

> "你的每一口，都为你量身定制。"

### 2.3 产品范围

**包含：**
- 用户健康画像构建（减肥、备孕、增肌、控糖、过敏管理等场景）
- "外面吃"路径：周边餐厅 + 菜品智能推荐（匹配健康需求）
- "自己做"路径：AI 生成个性化菜谱 + 食材清单
- 一键跳转多多买菜 / 叮咚买菜 / 美团买菜完成采购
- 饮食记录与反馈（吃得怎么样 → 推荐越来越准）

**不包含（v1.0）：**
- 自有供应链或自营配送
- 社交功能（社区/UGC菜谱分享）
- 营养师在线咨询
- 智能硬件接入（体脂秤、血糖仪等）
- 企业团餐/员工健康管理 B2B 场景

### 2.4 关键设计决策

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 自建 vs 跳转购物 | A.自建商城 B.跳转第三方 | B | 轻资产启动，利用现有生鲜平台履约能力，聚焦推荐引擎核心价值 |
| 推荐方式 | A.标签匹配 B.AI大模型 | B | 大模型能理解模糊需求（"想吃点暖胃的"），标签匹配无法覆盖长尾场景 |
| 盈利模式 | A.用户付费 B.平台佣金 | A+B | 基础功能免费获客，高级推荐付费；生鲜跳转走CPS分佣 |
| 首发平台 | A.小程序 B.原生APP | A | 微信生态获客成本低，分享裂变天然，先验证PMF再迁移原生 |

## 3. 用户画像

### 3.1 核心用户画像

#### 画像 A：减重奋斗者 · 小雅

> *28 岁，上海白领，BMI 26，尝试过 3 种减肥方法，每次坚持不超过 2 个月*

- **核心需求**：精准知道每顿能吃什么、去哪吃不踩雷
- **行为特点**：工作日外卖为主，周末偶尔做饭；经常在减肥社区找灵感但信息太杂
- **痛点**：点外卖时无法判断是否真的低卡；想做减脂餐但采购麻烦
- **使用频率**：每日 2-3 次（每餐前）

#### 画像 B：备孕准妈妈 · 小琳

> *32 岁，北京，计划 3 个月内怀孕，医生建议补充叶酸、铁、优质蛋白*

- **核心需求**：知道哪些食物/菜品富含备孕所需营养素，外出就餐也能吃得对
- **行为特点**：对食品安全敏感，愿意为品质付费
- **痛点**：餐厅菜单上找不到营养信息；自己做不知道搭配是否科学
- **使用频率**：每日 1-2 次

#### 画像 C：慢病管理者 · 老张

> *55 岁，轻度糖尿病，医生建议控制碳水、低GI饮食*

- **核心需求**：找到升糖指数友好的餐厅和菜谱
- **行为特点**：线下就餐为主，操作APP不太熟练
- **痛点**：外出吃饭完全靠猜，经常吃完血糖飙升
- **使用频率**：每周 3-5 次

### 3.2 用户旅程地图（核心路径）

```
[打开APP] → [选择健康目标] → [填写偏好/禁忌]
    → [主页：今日推荐]
        ├── "外面吃" → [定位] → [展示附近匹配餐厅+菜品] → [导航前往/跳转外卖]
        └── "自己做" → [AI生成菜谱] → [查看食材清单] → [一键跳转买菜APP] → [完成后饮食记录]
    → [历史记录 & 反馈] → [推荐越来越精准]
```

### 5.2 推荐引擎逻辑

```
推荐分数 = 
    w1 × 画像匹配度 (40%)
    + w2 × 场景匹配度 (25%)  
    + w3 × 社交信号 (20%)  // 评分、热度、同目标用户的选择
    + w4 × 多样性因子 (10%) // 避免重复推荐
    + w5 × 商业权重 (5%)  // 合作餐厅加权，未来可调
```

**画像匹配度计算：**
- 热量目标差（实际 vs 推荐）：±20% 以内满分
- 营养素匹配：蛋白质、碳水、脂肪比例
- 禁忌检查：含禁忌食材直接排除（硬规则）
- 偏好加权：历史好评菜系/口味加权

### 5.3 AI 菜谱生成 Prompt 策略

```
系统提示词：
你是一位专业营养师和厨师。请根据以下用户信息生成一道菜谱：

用户画像：
- 目标：{health_goal}
- 每日推荐热量：{daily_calories} kcal
- 饮食禁忌：{restrictions}
- 偏好菜系：{preferred_cuisine}
- 可用时间：{cooking_time}

生成要求：
1. 菜名（有吸引力）
2. 营养成分（热量、蛋白质、碳水、脂肪）
3. 食材清单（精确用量）
4. 烹饪步骤（5步以内，简洁）
5. 健康小贴士（为什么这道菜适合该用户）
```

### 7.1 技术栈建议

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| 前端 | 微信小程序原生 / Taro | 微信生态获客成本低，分享裂变天然 |

| 数据库 | PostgreSQL + Redis | 关系数据 + 推荐缓存 |
| AI服务 | 通义千问 / DeepSeek API | 菜谱生成 + 智能推荐排序 |
| 地图服务 | 高德地图 Web API | LBS定位 + POI搜索 + 导航 |
| 搜索 | Elasticsearch | 餐厅/菜品全文搜索 + 向量检索 |
| 云服务 | 阿里云 / 腾讯云 | CDN + OSS图片存储 + 短信 |
| 数据分析 | 神策 / GrowingIO | 用户行为分析 + 推荐效果追踪 |

### 7.3 关键技术挑战与对策

| 挑战 | 对策 |
|------|------|
| 餐厅菜品数据缺失 | MVP 阶段人工标注核心商圈餐厅；中期用大模型从菜单图片提取结构化数据 |

## 9. 边界与异常处理

### 9.1 异常场景矩阵

| 场景 | 触发条件 | 处理策略 | 用户提示 |
|------|---------|---------|---------|
| 定位失败 | 用户拒绝定位授权 | 允许手动输入地址，降级为城市级推荐 | "开启定位后可发现附近美食哦~" |
| 无网络 | 网络断开 | 展示缓存推荐内容 + 离线菜谱库 | "网络不太稳定，先看看缓存的内容吧" |
| 推荐无结果 | 附近无匹配餐厅/菜谱 | 放宽筛选条件（扩大范围/降低匹配阈值） | "周边暂时没找到完美匹配的，这些也不错~" |
| AI 生成失败 | API 超时/错误 | 降级到人工精选菜谱库推荐 | "菜谱正在路上，先看看精选推荐吧" |
| 买菜跳转失败 | 用户未安装/DeepLink失效 | 展示食材清单文本，支持复制 | "复制食材清单，打开买菜APP粘贴搜索即可" |
| 并发高峰 | 午餐/晚餐时段 | Redis 缓存热门推荐 + 限流 | 正常使用，后台降级策略用户无感 |
| 数据为空 | 新用户无历史 | 基于画像 + 热门推荐冷启动 | "为你精选今日热门推荐" |
| 权限不足 | 未登录/授权过期 | 限制个性化功能，基础浏览可用 | "登录后可享受专属推荐哦" |

## 10. 非功能性需求

### 10.1 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 菜谱生成响应时间 | < 2 秒 | AI 大模型调用需在 2 秒内返回结果，超时降级到精选菜谱库 |
| 推荐接口响应时间 | < 500ms | 基于缓存 + 预计算，保证主页推荐秒开 |
| 页面首屏加载 | < 1.5 秒 | 小程序包体积控制 + CDN 加速 |
| LBS 搜索响应 | < 1 秒 | 高德 POI 搜索超时降级到城市级推荐 |

### 10.2 准确性保障

| 保障措施 | 说明 |
|---------|------|
| 营养规则硬约束 | 热量、蛋白质、碳水、脂肪估算必须基于标准营养数据库，不允许 AI 自由发挥数值 |
| 营养师审核机制 | 备孕、减肥、控糖等特殊场景的 AI 生成菜谱，需经注册营养师审核通过后入库；审核不通过的菜谱标记为「人工精选」来源 |
| 过敏原强制检查 | 用户标注的过敏原（如花生、海鲜、麸质）在推荐结果中强制排除，零容忍 |
| 用户纠错机制 | 每道菜谱/菜品支持用户举报「营养信息不准」，累计 3 次举报自动下架并触发人工复核 |
| 医学免责声明 | 所有涉及健康建议的内容标注：「本内容仅供参考，不能替代专业医师或营养师意见」 |

### 10.3 隐私与安全

| 要求 | 说明 |
|------|------|
| 健康数据加密存储 | 用户 BMI、健康目标、饮食禁忌等敏感信息加密存储，传输使用 HTTPS + AES-256 |
| 位置信息最小化使用 | 位置信息仅用于推荐服务，不分享给第三方广告平台；支持用户清除位置历史 |
| 遵循《个人信息保护法》 | 隐私协议明确告知数据用途；支持用户导出全部个人数据、注销账号并删除所有数据 |
| 第三方 SDK 管控 | 高德地图、买菜平台跳转等第三方 SDK 的数据调用需在隐私协议中披露，并允许用户选择退出 |
| 饮食记录本地缓存 | 无网络时饮食记录优先本地缓存，联网后同步；缓存数据加密 |

### 10.4 可用性标准

- 核心路径（推荐 → 查看 → 跳转）可用性 ≥ 99.5%
- 买菜平台跳转失败率 < 20%（通过多平台兜底 + 复制清单保障）
- AI 大模型 API 故障降级策略：自动切换备用模型或降级到人工精选菜谱库，用户无感知

| 竞品快速跟进（美团/饿了么加健康标签） | 高 | 中 | 先发优势做个性化画像数据壁垒，聚焦细分场景 |
| 用户增长不及预期 | 中 | 高 | MVP 快速试错，必要时 pivot 方向 |
| 餐厅数据获取成本高 | 高 | 中 | 大模型提取菜单图片数据，众包用户贡献 |
